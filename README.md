# Sai Sujan S — Portfolio (Next.js)

This is the Next.js 14 (App Router) rewrite of the original Vite + React SPA, migrated specifically to
maximize search and AI-crawler visibility.

## Why Next.js + static export

`next.config.mjs` sets `output: 'export'`. That means `npm run build` produces a fully static
site (an `out/` folder of plain HTML/CSS/JS) — every section of the page is pre-rendered to real
HTML at build time. This is the single biggest SEO change from the old Vite SPA:

- **Old (Vite SPA):** the server sent an empty `<div id="root">`; all content only existed after
  React ran in the browser. Crawlers that don't execute JavaScript saw nothing. Even ones that do
  execute JS had to wait for it.
- **New (Next.js static export):** the full HTML for every section — headings, text, links — is
  already in the document Google/Bing/AI crawlers receive on the very first request. No JS
  execution required to see the content.

Interactive pieces (nav, forms, cursor effects, expand/collapse, scroll-reveal) are still React
Client Components and hydrate normally — the trade-off is only that they render as static markup
first, then become interactive, exactly like any modern framework's server rendering.

## Structure

```
app/
  layout.tsx        — Metadata API (title/description/OG/Twitter/robots/canonical), JSON-LD, font preloads
  page.tsx           — Home page, a Server Component that assembles every section
  globals.css        — Tailwind + all custom CSS (unchanged from the original project)
  robots.ts          — generates /robots.txt at build time
  sitemap.ts         — generates /sitemap.xml at build time
  manifest.ts        — generates /manifest.webmanifest at build time
  hooks.ts           — scroll-reveal, counter, and mouse-parallax hooks (client-only)
  components/        — one file per section; each has 'use client' only if it actually needs it
public/
  fonts/, favicon.ico, favicon.jpg, og-image.jpg, resume.pdf, llms.txt
```

### Server vs. Client Components
Only components that actually use hooks or browser APIs are marked `'use client'`:
`About`, `Contact`, `CustomCursor`, `Experience`, `Hero`, `LoadingScreen`, `Navbar`, `Projects`,
`SkillsSimple`, `RevealObserver`, `Footer`. `Education` and `EcommerceExpertise` are plain Server
Components — they ship zero client JS.

## What changed vs. the Vite version (besides the framework)

- **Fixed a hydration bug**: `Experience.tsx` read `window.matchMedia` directly during render,
  which is `undefined` on the server and could differ on a mobile client — this causes a real
  React hydration mismatch under SSR/static export (harmless in a pure client-rendered SPA, but not
  under Next.js). Moved it into `useEffect`.
- **Fixed a real content bug**: the "Resume" button and the sitemap pointed at two different PDFs
  (a 72KB real resume vs. an unrelated 4.5KB placeholder). Consolidated to one canonical
  `/resume.pdf` so the file people download matches the one search engines can index.
- `import.meta.env.VITE_*` → `process.env.NEXT_PUBLIC_*` (see `.env.example`).
- Removed the unused `@supabase/supabase-js` dependency (never imported anywhere).
- Metadata, robots.txt, sitemap.xml, and the manifest are now generated by Next's typed metadata
  routes instead of hand-maintained static files, so they can't drift out of sync with the code.

## Setup

```bash
npm install
cp .env.example .env.local   # fill in your EmailJS IDs for the contact form
npm run dev                  # local dev server
npm run build                # produces the static site in /out
npm run typecheck
npm run lint
```

## Deployment

`npm run build` outputs a static site to `/out`. Any static host works:

- **Render**: change the service type to a **Static Site**, build command `npm run build`,
  publish directory `out`. (A Node/Express "Web Service" is no longer needed — that's Node hosting
  for a site that doesn't need a server.)
- **Vercel**: works with zero config; Vercel also supports Next.js features that need a server, if
  you ever remove `output: 'export'` for dynamic routes later.
- **Netlify / Cloudflare Pages / GitHub Pages**: point the publish directory at `out`.

If you later add something that genuinely needs a server (an API route, on-demand ISR, etc.),
remove `output: 'export'` from `next.config.mjs` and deploy to a Node-capable host (Vercel, Render
Web Service) instead.

## A note on rankings

Everything in this repo is what's actually controllable through technical SEO: crawlable static
HTML, correct metadata, valid structured data, fast Core Web Vitals, and clean semantic markup.
None of that — nor anything else — can guarantee a #1 ranking; that also depends on backlinks,
domain authority, competition, and factors outside any codebase. This gets the technical ceiling
as high as it can go for a portfolio of this kind.
